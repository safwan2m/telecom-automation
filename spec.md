# Telecom Network Automation — Project Specification

## High-Level Idea

Build a multi-agent Agentic AI system that accepts deployment parameters for a specific geographic area — Malleswaram, North Bangalore — and automatically plans, configures, deploys, or modifies an Open RAN (O-RAN) compliant 4G/5G network. The system is context-aware of the live deployment and continuously optimizes for load balance and operator efficiency. The live deployment is simulated by 3 DU containers and 1 CU container covering a population of 40,000 residents + 15% commuter overhead (46,000 effective), with 18,400 peak active UEs at 40% operator market share. All UE telemetry is synthetic, generated by the DU simulators using realistic hardware specifications and diurnal load curves.

- **Orchestrator** — Understands operator commands via an LLM and routes them to the right tools; streams the response back to the operator.
- **Controller** — The single source of truth for the network topology; handles live cell/DU moves and topology CRUD.
- **Planning Agent** — Designs new network layouts for cell placement and DU/CU grouping using placement algorithms, PCI planning, and slice allocation. UE counts and hardware limits are calibrated to real market equipment specs.
- **KPI Agent** — Continuously monitors cell health using a bidirectional LSTM model, detects anomalies (overload, poor signal, power waste), and takes autonomous corrective actions.
- **Map Server** — A live map container showing the current cell layout with per-cell KPI overlays, colour-coded by vendor and generation.

The RAN is a 4G/5G NSA mix — both 4G LTE and 5G NR connect to the same NSA core (shared AMF/SMF/UPF). 5G NR deployment uses split CU/DU/RU architecture throughout. The KPI agent, controller, or planning engine can reorganise cells at any time in response to changing load or operator instruction.

### Network Topology — Malleswaram, North Bangalore

**Population model:** 40,000 residents + 15% commuter overhead (railway station transit hub) = 46,000 effective. Tier-1 operator 40% market share → **18,400 peak active UEs**. Busy-hour traffic = 18,400 × 0.025 Erlangs/user = **460 Erlangs**. At 16 Erlangs per sector (Erlang-C, 2% blocking) → **29–30 sectors required**.

| CU | DU | Sites served | Cells |
|---|---|---|---|
| CU-MLS | DU-MLS-1 | RWS, 18C, BEL, SNK (north) | 12 |
| CU-MLS | DU-MLS-2 | SPG, 3MN, 10C (central) | 9 |
| CU-MLS | DU-MLS-3 | MGR, CHD, 6CR (south-west) | 9 |

**30 cells (10 macro sites × 3 sectors). 700 MHz (n28) not deployed — coverage radius (~8.4 km) extends beyond Malleswaram to Peenya. Sector mix per site:**

| Sites | Sector 1 | Sector 2 | Sector 3 |
|---|---|---|---|
| RWS, 18C, SNK, SPG, 10C (high traffic) | 5G n78 3500 MHz | 5G n41 2500 MHz | 4G B3 1800 MHz |
| BEL, 3MN, MGR, CHD, 6CR (residential) | 5G n78 3500 MHz | 4G B40 2300 MHz | 4G B3 1800 MHz |

**Cell-level max UEs: 16,500 (900×10 n78 + 700×5 n41 + 300×5 B40 + 250×10 B3). active_ues_peak: 18,400 (small headroom gap absorbed by Erlang-C blocking tolerance).**

### Vendor Distribution (25% each — 10 cells per vendor)

|---|---|---|---|---|---|

## Reference

### A. Input Parameters

Parameters accepted by the planning engine and orchestrator to deploy or reorganise a network:

| Parameter | Description | Example |
|-----------|-------------|---------|
| `hardware_resources` | Radio hardware model + capability | `Nokia AirScale MAA 64T64R` |
| `geographic_area` | City zone or GeoJSON polygon | `"ITPL & EPIP zone, Whitefield"` |
| `expected_user_density` | Users per km² | `50000` |
| `traffic_profile` | Slice mix + peak hour | `{"eMBB":0.7,"URLLC":0.2,"mMTC":0.1,"peak_hour":19}` |
| `fiber_availability` | Fiber map or site list | `["Koramangala","Whitefield",...]` |
| `spectrum_bands` | Licensed bands | `["n78","n28","B3","B40"]` |
| `latency_constraints` | E2E and fronthaul targets | `{"e2e_ms":10,"fronthaul_us":100}` |
| `compute_resources` | Per-site server capacity | `{"cpu_cores":32,"ram_gb":64}` |
| `deployment_budget` | CAPEX/OPEX envelope (USD) | `2000000` |

All planning fields are required. The tool schema uses `"required": []` so the LLM never infers missing values; the planning server returns a structured missing-fields response (HTTP 200) and the orchestrator relays the list to the operator in the next turn.
