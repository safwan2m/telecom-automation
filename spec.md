# Telecom Network Automation вҖ” Project Specification

## High-Level Idea

Build a multi-agent Agentic AI system that accepts deployment parameters such as radio hardware resources, geographic area, expected user density, traffic profile, spectrum bands, latency constraints, compute resources, and deployment budget, for a specific geographic area вҖ” Malleswaram, North Bangalore вҖ” and automatically plans, configures, deploys, or modifies an Open RAN (O-RAN) compliant 4G/5G network. The system is context-aware of the live deployment and continuously optimizes for load balance and operator efficiency. The live deployment is simulated by 3 DU containers and 1 CU container covering a population of 40,000 residents + 15% commuter overhead (46,000 effective), with 18,400 peak active UEs at 40% operator market share. All UE telemetry is synthetic, generated by the DU simulators using realistic hardware specifications and diurnal load curves. 
- Orchestrator вҖ” Understands operator commands with the use of an LLM and routes them to the right tools and gives appropriate response back to the operator. 
- Controller вҖ” the single source of truth for the network topology; handles live cell/DU moves
- Planning Agent вҖ” designs new network layouts for cell placement and DU/CU grouping or if at all new cell placement is required for the increasing demand, UE counts and hardware limits are calibrated to real market equipment specs using placement algorithms, PCI planning, and slice allocation.
- KPI Agent вҖ” Continuously monitors cell health using a bidirectional LSTM model, detects anomalies (overload, poor signal, power waste), and takes autonomous corrective actions
- Map Server вҖ” A live map container shows the current cells layout with per-cell KPI overlays, colour-coded by vendor and generation.

 The RAN is a 4G/5G NSA mix вҖ” both 4G LTE and 5G NR connect to the same NSA core (shared AMF/SMF/UPF). 5G NR deployment uses split CU/DU/RU architecture throughout; 

The KPI agent, controller, or planning engine can reorganise cells at any time in response to changing load or operator instruction. Input may be partial вҖ” the system will request missing fields or flag redundant ones.

## Agent Architecture

### Agent 1 вҖ” LLM Orchestrator (`agents/orchestrator/`)
- FastAPI on port 8082
- `POST /chat` вҖ” natural language вҶ’ Gemini tool-calling вҶ’ streaming response
- Tools: `query_network`, `list_cells`, `query_cell`, `move_cell`, `move_du`, `plan_network`, `plan_network_multi_period`, `apply_plan`, `get_alerts`
- Injects live network snapshot from Controller into every system prompt

### Agent 2 вҖ” Controller (`agents/controller/`)
- FastAPI on port 8080 вҖ” single control plane / source of truth
- `GET /network` вҖ” full topology + live KPIs merged in one response
- `GET /cells`, `/dus`, `/cus` вҖ” component listings with KPIs
- `POST /move/cell`, `/move/du` вҖ” live topology changes (atomic write to topology.json)
- Topology changes propagate to DU/CU simulators within 5 s via file-watch polling

### Agent 3 вҖ” Planning Engine (`agents/planning/`)
- FastAPI on port 8081
- `POST /plan` вҖ” takes deployment parameters, returns complete network plan; set `use_mip=true` for MIP-optimal placement
- `POST /plan/multi-period` вҖ” multi-period MIP planning (Case A: phased rollout, Case B: diurnal/event demand shift)
- `GET  /demand-clusters` вҖ” list pre-defined Bangalore demand clusters and preset period profiles
- Algorithms: density-weighted heuristic (default) OR MIP-optimal placement (Almoghathawi et al. 2024), graph-coloring PCI planner, PRB slice allocator, distance-based fronthaul/midhaul router
- `POST /plan/apply` вҖ” pushes accepted plan to Controller as topology update

#### Planning algorithms вҖ” propagation models

| Model | Used by | Environment | Notes |
|---|---|---|---|
| COST-231-Hata | DU/CU simulators, Map Server | Urban macro, empirical | Coverage radius, KPI simulation |
| **COST-231-Walfisch-Ikegami** | **MIP placer** | **Urban NLOS** | **Path loss for link-budget feasibility and SINR constraints** |

#### MIP-based placement (Almoghathawi et al., 2024)

Reference: Almoghathawi Y., Bin Obaid H., Selim S. вҖ” *"Optimal location of base stations for cellular mobile network considering changes in users locations"*, Journal of Engineering Research 13 (2025) 561вҖ“567. DOI: 10.1016/j.jer.2024.04.020

**Formulation** вҖ” Mixed integer programming minimising total network cost:

```
Minimise  ОЈ_j ОЈ_t ( c_jtВ·z_jt  +  r_jtВ·y_jt )
```

where `z_jt` = build BS at site j in period t (one-time CAPEX), `y_jt` = BS active in period t (per-period OPEX).

Subject to:
- **(2) Single-build**: each candidate site built at most once across all periods
- **(3) Coverage**: every demand cluster served by a BS built in this or an earlier period
- **(4) Activation**: BS can operate only after it has been built
- **(5) Unique assignment**: each demand cluster assigned to exactly one BS per period
- **(6) Implies-active**: assigning demand to a site forces that site active
- **(7) Capacity**: channel demand at each BS вүӨ Оҙ_j (max UE capacity)
- **(8) SINR QoS**: received SINR at each demand cluster вүҘ SINR_min; constraint linearised as:
  `Оұ(i,t)В·(1 + SINR_lin) вүҘ SINR_linВ·P_noise + SINR_linВ·ОІ(i,t)`

**Demand node concept** (Tutschku 1998): traffic in each area is represented as a finite set of *demand clusters* D. Each cluster has a channel requirement ПҒ_i. Demand clusters are separate from candidate sites S; 10 pre-defined Bangalore clusters, one per area.

**Multi-period modes**:
- **Case A вҖ” permanent/expanding**: each period adds new demand clusters (phased network rollout). BSs built in period 1 can serve demand in periods 2 and 3, avoiding redundant infrastructure spend.
- **Case B вҖ” temporary/shifting**: demand clusters shift between periods (diurnal patterns: residential peak вҶ’ IT-hub business hours вҶ’ evening commute). The solver minimises total cost across all scenarios.

**Cost model**:
- `install_cost` (c_jt): one-time CAPEX; incurred once when the site is built
- `op_cost` (r_jt): per-period OPEX; incurred every period the site is active
- Both vary by location and time period (configurable)

**Solver**: CBC (via `pulp`), open-source MIP solver. Falls back to heuristic placement on timeout or infeasibility.

### Agent 4 вҖ” KPI Monitoring Agent (`agents/kpi_agent/`)
- Background process (no exposed port)
- Polls InfluxDB every 30 s; maintains 6-step (60 s) sliding window per cell
- Bidirectional LSTM classifier (5 classes): NORMAL, OVERLOAD, UNDERLOAD, SINR_LOW, POWER_WASTE
- Rule-based fallback for first 60 s while window fills; then AI inference with 70% confidence gate
- Actions: calls `/move/cell` for overload rebalancing; writes `alerts` to InfluxDB

### Agent 5 вҖ” Map Server (`agents/map_server/`)
- FastAPI on port 8083
- Serves a Leaflet.js interactive map of all 30 cells in Malleswaram
- `GET /` вҖ” HTML map page (auto-refreshes every 30 s)
- `GET /api/cells` вҖ” proxies Controller `/network`, returns GeoJSON-ready cell list
- Colour-coded by vendor (Nokia=blue, Ericsson=green, Samsung=purple, ZTE=orange)
- Visual status: solid = 5G NR, faded = 4G LTE; red fill = overloaded (PRB > 85%), amber = SINR < 5 dB
- Click popup: vendor, hardware model, band, DU/CU, PCI, connected UEs, PRB, SINR, RSRP, power, throughput

## Known Gaps (as of current implementation)

| Gap | Location | Impact |
|---|---|---|
| `/topology/replace` missing | `controller.py` | `apply_plan` cannot deploy a new plan to the live network; falls back to returning topology JSON |
| `plan_to_topology()` drops cell fields | `planner_api.py:244` | Applied plans lose vendor, generation, antenna_config, tx_power_w вҖ” DU simulators fall back to wrong defaults |
| RACH procedures not implemented | `du_simulator.py` | No preamble, RAR, MSG3/MSG4 or contention modelling; UEs jump directly from pool to connected state |
| No Grafana dashboards provisioned | `dev-env/grafana/` | Grafana starts with InfluxDB connected but no panels; time-series KPI views must be built manually |
| No Neighbor Relation List (NRL) | topology.json, controller | Neighbor queries and ANR-based SON optimization not possible |
| Per-UE query not exposed via chat | `tools.py` | `ue_usage` and `ue_mobility` are written to InfluxDB but no tool exposes them to the LLM |
| SON corrective actions limited | `kpi_agent.py` | Only OVERLOAD triggers an autonomous action (cell move); SINR_LOW, POWER_WASTE, UNDERLOAD generate alerts only |
